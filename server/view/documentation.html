Introduction

Obesity Prevalence Visualizer is a web application for data visualization. The purpose of this page is to visualize both data and statistics about obesity in USA. We created statistics charts importing data using an API REST service. You can visualize data from different states, categories (total, age, gender, income, education, ethnicity) and compare values from different states, categories and years and export charts in csv, WebP and svg format. You can also create an account on our web application.

Basic functionalities of our web application

The purpose of our web application is obesity data visualization. On our web application you can navigate as user or guest. Our web page has multiple articles about health and wellness, diseases and conditions and obesity statistics in USA. For visualizing statistics you need to go on our statistics visualizer page where you can select the states , the chart type and series value you want to see. You can also select more states or series values to compare them. We have four types of charts, map, line chart , bar chart and horizontal bar chart, for all on them you can hover and see the exact value you want to see, also you can hover the legend for bar chart and line chart to see only the bar or line you want to see. You can export all our chats in cvs, svg and WebP format.


You can create an account on our web application and log in. For visualizing your personal data you can go on your profile page where you can change your personal info, first name, last name, email and password. If you have an admin account, from your profile you can also manage user access and add statistics data.


To create our web application front-end we used HTML, CSS and JavaScript. We used HTML to describe the structure and the meaning of the website, CSS to describe a web page's appearance/presentation and JavaScript to describe functionality and behavior.


For creating the charts we used D3.js, a JavaScript library for manipulating documents based on data. D3 helps you bring data to life using HTML, SVG, and CSS. We started by sending a network requests to the obesity api service and loading new information using JavaScript Fetch. After receiving the data we create the SVG and defining the scales for the chart and add axes and labels to the SVG. Next, we created bars/lines corresponding to the data values.


To export the chart into svg format we encoded the SVG as a Base64 Image.and then we created the link which we set to the href of the svg. The data URI scheme consists of data:[media type][;base64],data. Since we know that the media type is image/svg+xml and that we want base64 representation of resulting data, we put in the href value data:image/svg+xml;base64 to which we concatenated base64 representation of svg. To export as a WebP we created a canvas and an img and set the src with the value data:image/svg+xml;base64 to which we concatenated base64 representation of the svg. To export data as a csv we concatenated all the chart info into a variable. Since we know that the media type is text/csv and that we want charset=utf-8 representation of the data, we put in the href value data:text/csv;charset=utf-8 to which we concatenated encodeURI representation of chart info.

User experience

We tried to create a web application that provide meaningful and relevant experiences to users. Our main purpose regarding user experience was to create a useful and usable application. We tried to offer our clients more info about obesity, not only statistics. Our web application is easy to use and desirable. We tried to anticipate what users might need to do and ensuring that the interface has elements that are easy to access, understand, and use to facilitate those actions. Users can use the burger navigation menu, header and footer to navigate to any other page from the current one. We focused on how information is organized, structured, and presented to users, so we have three big categories of information, statistis, articles about health and wellness and articles about diseases and conditions


Our visual design is simple and aesthetically pleasing. We used the same template presentation for the html pages using the same color pallete and structure with the header, menu, sections and footer.

Design motivation

We created a web application with a simple interface. Our design is consistent and use common UI elements (header, footer, navbar and common section for similar types of pages). We create a pattern in layout and design to help facilitate efficiency. Once a user learns how to do something, they should be able to transfer that skill to other parts of the site. We strategically used colors to direct attention toward or redirect attention away from items using color, light, contrast, and texture to your advantage. We mainly used different types of blue and grey to keep the consistency.

Overall architecture

We decided to usse a Service-Oriented Architecture for the Obesity Prevalence Visualizer. We created more services for our main application. This approach allows us to make use of services available in the network and integrate them into a coherent and system.


This kind of service is beneficial because the services are designed as components and can be reused more effectively, thus reducing development time. Also, a service is completely defined by its own documentation. They hide their logic, which is encapsulated within their implementation. As services are independent of each other they can be updated and modified easily without affecting other services. They have control over the logic they encapsulate and, from a service consumer point of view, there is no need to know about their implementation. Moreover, its facilities are easily available to anyone on request and are more reliable because it is easy to debug small services rather than huge codes.

Our main page application is running at localhost:3000 and it’s the service consumer. Our REST APIs are represented by the Accounts API and Data Statistics API which are the service provider. As it can be seen in the diagram the users access our frontend server through web from a browser. Then the front server sends requests to our REST APIs accessing their routes with their controllers and querying the database so that a final response can be formed.


After the frontend was ready, we started the services by creating our APIs. We implemented a REST API, a Representational State Transfer Interface, using the Nodejs Technology with MongoDB Database. Our RESTful API uses existing HTTP methodologies and the communication is done through JSON responses. We use GET to retrieve a resource; PATCH to change the state of or update a resource; POST to create a new resource; and DELETE to remove it. The status codes we sent are also representative for the response we send back. We use 400 - BAD REQUEST when there are errors. When there is a successful response, we send 200 - OK for GET requests, 201 - CREATED for POST requests, 204 - NO CONTENT and UPDATED for DELETE and PATCH responses.

The first API we created was the Statistics Data API. This is running at localhost:3001 and is used for handling the data manipulation for the statistics. We mostly make GET requests from frontend, to the data so we can create the charts, but we can also add new data for the existing countries or for upcoming years using POST request. We can modify the data from the database using PATCH request or even remove data from database with a DELETE request.


The next API we developed was the Accounts API, running at localhost:3002 and used for handling the user authentication on the main web application. When a user wants to register, we use a POST method to database where we save his email and encrypted password so that he can log in after registered successfully. When a user wants to sign in, we generate a unique token which we set in the Local Storage only if the email and password he entered are a match in the auth_management database. We can also perform a log out for a user, moment in which the token corresponding to his session is sent in the request body and we set the token to null. A logged in user can add or change information from his profile such as: he can add his first name, last name, change his email or even his password. These operations make a POST request each time to the Account API sending the token in the request body. If the logged in user is an admin, he can send requests to the server to restrict/unrestrict normal users based on their email address, so that those users will not be able to log in in the future. Of course, we can also use the API to GET all users or a single logged in user based on his token and to DELETE all users.

Data Structure

The first database we created was the obesity_usa database in which we stored data in more collections based on its category.

We exported data from BRFSS in csv format. We formatted the data so that we kept only the essential information and saved it in csv and json format. Then we imported the documents in our database and we formatted the data again so that would have the same structure in all collections. That made the work with those data easier. The fields that we access the most are Description representing the year, LocationDesc representing the name of the state, Data_value representing the obesity percentage and Stratification1 representing the subcategory of the collection.


The next database we created was the auth_management database with the users collection in which we store the information about all users that register on our web application. A user can have the userType : user or admin. A user can register with an email and encrypted password and then he can log in with those credentials. He can log out, modify first name, last name, email and even the password. When a user is logged in he is identified by his token. If the user is admin he can set the restricted field of a user to true or false, and that specifies if that user can log in or not anymore.


We also payed attention to NoSQL Injection. NoSQL Injection happens when a user enter an input which with we will make a database query, but that input is not necessary the input we thought of, it’s usually something that want to access our all database info or even modify it. An example would be when we query the database to find a document by its token. If the token we receive looks like this: “token”: { “$gte" : ”" }, then instead of returning only the user with a token it will return all users from database. So, to prevent that kind of attack we always verify if the token we get is a valid token with a regex. We do the same for the email so there will not exist malformed email addresses.

Code management

We used Github for our code management: https://github.com/andrasimion99/twProject . This is the repository in which we both, Filimon Raluca and Simion Andra, pushed our code in the master branch.


For the frontend code, we modularized it such that we have the folder /views with all our .html pages. There is also the /resources/css folder within which we find all the .css files and a folder /img with the images we used in our web application. Moreover, we can find the /scripts folder which contains all the .javascript files we used, separated in different folders based on their purpose: /statisticsUtil, /userUtil, /adminUtil, all kind of charts etc.


As for our APIs, we modularized our code in separated folders and files respecting the MVC structure. So, we have a model folder representing the schema with its methods for the resources from the database. Also, we have the router folder in which we have the connection to database, we parse the request URL, the query parameters and the request body depending on the method received. Also, the router folder contains a file in which we specified each path and method to a handler. The /controllers folder contains all controller classes that have the handlers methods for requests. The config folder contains all configuration and constants such as the PORT for the service and the MONGO_CONNECTION_URL.


We also resorted to the use of tests for the APIs so that we insured the requests we make are returning the expected response both for the Accounts API and the Data Statistics API. We tried making unit tests for GET/POST requests making a call to the API with http and checking the response with assert.

Response structure

The structure of the API responses' body is as follows:

  • for successful responses, a JSON object containing the properties:
    • status: success
    • data: An object, structure detailed for each route below.
  • for unsuccessful responses, a JSON object containing the properties:
    • status: fail
    • error: An object containing a message property, and sometimes additional helpful properties.
